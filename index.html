The Good

JSM - Javascript Simple Modules

Started back in 2009, the goal was to bring structure to the way we build sites.

Used on LG.com, Marlboro.com, and many others.

It's as much a technical achievement as a philosophy toward coding for our requirements.

I'm immensely proud of the rising tide effect this had on our team globally.

The Tech

Modules

I learned a great deal from Brian Cavalier, John Hann and Alex Sexton.

  1. Well-structured applications demand a modular approach
  2. Decouple (or loosely couple) your modules from their dependencies and each other
  3. Usage of a mediator to wire interoperability ensures no spaghetti code

I had a bright idea

Use Classes as Modules

I found the John Resig Class and used that as a basis from my modules.

We now have some sugar for the prototype chain, options, an extend and _super method.

I made a Widget Class that contains some common properties and methods, creating the basis for everything UI.

Making new widgets was easy and each one came with a base set of functionality.

Elements as Instances

Instantiating a Class meant giving an element a css class for that widget.

HTML5 data attributes are used for options.

Configurable, flexible variations in a declarative way.

One foot out of jQuery

We used a bridge function to bind the objects to the elements.

We weren't writing jQuery plugins directly, but a set of methods that got turned into a plugin.

While jQuery was a hard dependency, it didn't define the architecture.

Advanced Usage

A few forks existed over the years that extended widgets based on client capabilities.

For example, we used a "touch/swipe" version of a widget for touch devices that extended a non-touch widget. No polyfills or hacky overrides needed.

The People

Developers

Our organization suffered from a creeping ignorance.

A general "download-the-latest-plugin-and-implement-the-api" culture was forming with the explosion of a million open-source projects.

JSM made our team stop and think about how to write javascript for the defined requirements.

Our code was cleaner, clearer, and easier to document, understand, extend, and share.

Roll your own

I taught the organization to use css3 for everything in the modules. No css or text allowed.

This way we reduced battery consumption, used the GPU and developed easier and cleaner code.

Content Editors & Other Development Teams

This setup had a huge benefit to content editors, implementors, and folks who used the CMS that this was often put into.

Adding functionality was as easy as adding the script tag and some well defined markup.

If interop was needed, it was a piece of cake.

The Bad (sort of)

Responsive+ & RPlusBuildScript

Caveat Emptor

Responsive+ is a great solution.

I still believe UA detection has a place in responsive design.

The tool made R+ development easier and was genuinely useful.

In a nutshell...

You can read about Responsive+ here.

But Lessons were Learned...

After Shiva Vannavada and I created the idea of Responsive+, I set about to create a build script in nodeJS.

Built before Grunt was a thing, a custom node script was made.

Then GruntJS came out.

NPMs were flourishing.

But other issues were more systemic...

Build profiles were too rigid.

Desktop, Tablet, and Mobile are too narrowly focused.

Device Atlas (our server solution) had huge gaps.

Css structure was complex...

Css selectors were shared across multiple files

Base.css held shared properties...

Mobile.css held mobile properties...

Desktop.css held desktop properties...

Tablet.css held tablet properties...

...all for the same class.

Did I mention we chose Less.js

Right before it looked to become adanbonware?

Templating was also complex...

The delineation between where css should begin and templates should end was not always clear.

Needlesstosay...

This was hard to grok for some devs.

However

Despite the complexity and missteps...

Nothing went down the wire that wasn't needed for that client.

My obession with handing mobile network latency was satisfied.

The client was happy

and it worked as expected.

Now where are we?
